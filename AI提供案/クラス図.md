## このクラス図で重要な設計ポイント

### ① Recorder は Viewer を知らない

```text
Recorder
    ↓
ForwardRecord(.pt)

Viewer
    ↑
ForwardRecord(.pt)
```

Recorder は「保存するだけ」、Viewer は「表示するだけ」です。

---

### ② ImageRenderer は Strategy パターン

Viewer は

```python
renderer.render(tensor)
```

しか呼びません。

そのため、

* RGBRenderer
* MeanRenderer
* MaxRenderer
* HeatmapRenderer
* GradCAMRenderer（将来）

などを簡単に追加できます。

---

### ③ Viewer はレイアウトだけを担当

OverviewViewer

```
□ □ □ □ □
```

FeatureMapViewer

```
□□□□□□
□□□□□□
```

という**並べ方**だけを知っています。

画像の生成方法は知りません。

---

### ④ Trainer が保存タイミングを管理

Recorder は

```python
recorder.save(...)
```

しか知りません。

Trainer が

* 何epochごとか
* どのサンプルか

を決定します。

## 私がクラス図を見て気になった点（実装前に修正したい）

現在の設計では **`LayerRecord` が入力テンソルと出力テンソルを直接保持**しています。

これはシンプルですが、将来的に

* backwardの勾配
* 重み勾配
* attention
* activation statistics
* hookで取得した任意の値

などを追加すると、`LayerRecord` に属性がどんどん増えてしまいます。

そこで、私は **`LayerRecord` を「レイヤーの観測データを格納するコンテナ」として設計する**ことを提案します。

例えば、

```python
class LayerRecord:
    layer_name: str
    layer_type: str
    data: dict[str, torch.Tensor]
```

とすれば、

```python
layer.data["input"]
layer.data["output"]
```

から始めて、将来的には

```python
layer.data["grad_input"]
layer.data["grad_output"]
layer.data["attention"]
```

のように自然に拡張できます。

**初期実装では `input` と `output` しか使いませんが、設計としては長期的に保守しやすくなります。**

この変更は現在の仕様と矛盾せず、将来のResNetやTransformer対応にも役立つと考えています。
